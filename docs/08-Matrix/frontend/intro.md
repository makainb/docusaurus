## 1、名称的由来

**Matrix** 是一个 **架构**，而不是一个 **框架** 。名字源于黑客帝国的电影 **矩阵** ，后续所有的一切都被 **Matrix** 所笼罩。

## 2、架构初衷

我为什么要搞 Matrix 架构呢？

好吧别猜了，建立架构的目的是**代码和UI风格统一**、**项目可扩展**、**组件可复用**、**底层可维护**，

说人话，核心目的只有一个：**提升开发效率、提效开发质量**。


## 3、设计原则

| **目录名称**      | **设计原则 (Design Philosophy)**                             | **核心职责**                                                 |
| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `utils`      | **核心工具与无状态封装**。单例、无状态、全局唯一。 | 这些工具不依赖 Vue 的 setup 环境，在任何 .js 或 .vue 文件里 import 就能用。 |
| `hooks`      | **响应式逻辑与状态管理**。逻辑复用，自动化状态处理。 | 将“脏活累活”抽离组件，保持 Vue 文件纯净。 |
| `plugins`    | **外部能力接入与初始化**。能力注入，一次配置，全局生效。   | 它们通常在 `main.js` 中通过 `app.use()` 挂载，是架构的“插件接口”。  |
| `components` | **增强型基础组件**。透明代理，UI 标准化。 | 为了保持 UI 高度统一，子应用禁止直接调用原生 Element。   |
| `styles`     | **视觉一致性**。定义全公司的“工业美学”标准。                 | 全局 CSS 变量（主题色、圆角）、Vxe-Table 的统一样式覆盖。    |
| `theme`     | **皮肤**。定义全公司的“工业美学”标准。                 | 全局 CSS 变量（主题色、圆角）、Vxe-Table 的统一样式覆盖。    |


## 4、流程分解
| **目录**         | **对应动作**     | **给开发者的价值**                                       |
| ---------------- | ---------------- | -------------------------------------------------------- |
| `utils`      | 调接口、开遮罩   | 屏蔽网络层复杂性，统一报错处理。                         |
| `plugins`    | 换语言、配表格   | 抹平第三方库的配置差异，实现开箱即用。                   |
| `hooks`      | 定义请求、算状态 | **最核心：** 减少 70% 的 `ref` 定义和 `try-catch` 代码。 |
| `components` | 写页面、做交互   | **最直观：** 拖拽式开发，界面风格秒变专业。              |


## 5、样例讲解

我用一个**工单提交**的例子来说明：

1. **用户点击** `components/MkButton`。
2. **业务逻辑** 调用 `hooks/useRequest` 发起 POST。
3. **`useRequest`** 内部通过 `utils/loading.js` 开启遮罩，并调用 `utils/http.js` 发送请求。
4. **`utils/http.js`** 发现 Token 过期，通过 `plugins/i18n.js` 获取翻译后的错误提示。
5. **`useRequest`** 拿到结果，自动关闭遮罩，通知业务代码更新数据。


:::tip[写在最后]
我做的不是一个简单的工具包，而是一套生产线。让大家从“搬砖头”（写重复配置）进化到“开吊车”（调用标准指令），这才是前端中台的意义。


设计的“三层境界”：
- 能用：提供了 http 和 pinia。
- 好用：不需要 JSON 转换，不需要处理响应式丢失。
- 爽用：快捷属性访问、请求与 UI 联动、一键批量操作。
:::
